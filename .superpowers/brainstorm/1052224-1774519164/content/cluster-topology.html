lw-c1 Cluster Topology

16GB RAM / 256GB NVMe — pick a VM layout that fits

A

Lean: 1 Control Plane + 1 Worker

VMs: CP (2 cores, 3GB) + W1 (2 cores, 6GB) + Proxmox (2GB) = 11GB used

Disk: CP 40GB + W1 60GB + template 32GB + OS 20GB ≈ 152GB

Workloads on W1: n8n workers, GitHub Actions runners

Pros

  • 5GB headroom for growth
  • Simple, reliable
  • All workloads on dedicated worker

Cons

  • No worker redundancy
  • Single point of compute failure
B

Balanced: 1 Control Plane + 2 Workers

VMs: CP (2 cores, 3GB) + W1 (2 cores, 4GB) + W2 (2 cores, 4GB) + Proxmox (2GB) = 13GB used

Disk: CP 40GB + W1 50GB + W2 50GB + template 32GB + OS 20GB ≈ 192GB

Workloads split: n8n workers on W1, GitHub runners on W2

Pros

  • Workload isolation
  • Worker redundancy via K8s scheduling
  • Room to scale per workload type

Cons

  • Only 3GB headroom
  • Tight on RAM under load
C

Compact: Control Plane doubles as Worker

VMs: CP+W (4 cores, 10GB, taints removed) + Proxmox (2GB) = 12GB used

Disk: CP+W 80GB + template 32GB + OS 20GB ≈ 132GB

Workloads: all on single VM (single-node K8s)

Pros

  • Most RAM per workload
  • Simplest to manage
  • Most disk headroom

Cons

  • No K8s node separation
  • Control plane + workload compete for RAM
  • Less realistic cluster experience

My recommendation: B (Balanced) — workload isolation is worth the tight headroom on a homelab that won't run all workloads at full capacity simultaneously.
