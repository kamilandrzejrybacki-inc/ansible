K8s Distribution for lw-c1

Single bare-metal Ubuntu node — which K8s flavour fits best?

A

K3s — lightweight single binary

Designed for single-node and edge. Installs in 30 seconds, runs as a systemd service. Built-in containerd, Flannel CNI, Traefik ingress.

Memory: ~500MB for K3s itself + workloads

Playbook: New k8s/k3s-setup/ — simple, ~50 lines

Pros

  • Lowest overhead of any option
  • Single binary, systemd-managed — survives reboots cleanly
  • vCluster works on K3s
  • Near-zero ops burden

Cons

  • Not vanilla K8s (minor API differences)
  • New playbook needed
B

kind — K8s in Docker (existing playbook)

Runs K8s nodes as Docker containers. Already implemented in k8s-full-setup with Helm, k9s, Headlamp, ArgoCD. Just retarget inventory to lw-c1.

Memory: Docker daemon + containers (~1-1.5GB overhead)

Playbook: Extend existing k8s-full-setup — change inventory + connection

Pros

  • Already fully built — Helm, k9s, Headlamp, ArgoCD included
  • Minimal new Ansible work
  • Easy to reset (delete + recreate cluster)

Cons

  • Docker-in-Docker for workloads is awkward (GitHub runners)
  • Doesn't survive reboots by default — needs extra setup
  • kind is designed for local dev, not persistent workloads
C

kubeadm — vanilla K8s bare metal

Production-grade K8s bootstrapped directly on Ubuntu. Uses the roles already written in proxmox-k8s-setup (k8s-prereqs, k8s-controlplane) — just remove the Proxmox VM provisioning step.

Memory: ~1.5-2GB for control plane components

Playbook: New k8s/kubeadm-setup/ — reuse existing roles

Pros

  • 100% vanilla K8s — no surprises
  • Roles already exist (k8s-prereqs, k8s-controlplane)
  • Systemd-managed, survives reboots
  • Easiest to extend to multi-node later

Cons

  • More memory than K3s
  • More complex bootstrap than K3s

My recommendation: A (K3s) for a single persistent node running real workloads. Survives reboots, lowest overhead, vCluster-compatible. The existing Helm/ArgoCD playbooks still apply on top.
