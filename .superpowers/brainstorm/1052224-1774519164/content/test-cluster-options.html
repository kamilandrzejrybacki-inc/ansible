Test Cluster Strategy

16GB RAM is already 13GB used — how to fit a test environment

A

vCluster — Virtual K8s inside the main cluster

Runs a full K8s API server as a pod inside your existing cluster. Looks and behaves like a real cluster from the outside, but shares the underlying nodes.

RAM overhead: ~200–400MB per vCluster

How: Deploy via Helm into a test namespace. Point your kubeconfig at it. Tear down and recreate in seconds.

Pros

  • Full API isolation (own namespaces, RBAC, CRDs)
  • Near-zero extra RAM
  • Ephemeral — perfect for CI/test teardowns
  • Already supported by Helm (which we're deploying anyway)

Cons

  • Shares worker node resources — noisy neighbour possible
  • Some cluster-level resources aren't fully isolated (nodes, storage classes)
B

Dedicated lightweight VM — single-node K3s

Drop prod to Option A (1 CP + 1 worker) to free up ~4GB, then add a small K3s VM for test.

Budget: Prod CP (3GB) + Prod W1 (6GB) + Test VM K3s (3GB) + Proxmox (2GB) = 14GB

Pros

  • Fully separate cluster — real isolation
  • K3s is lightweight, installs in minutes
  • Different K8s version possible for compatibility testing

Cons

  • Prod loses a worker node
  • Still tight on RAM
  • K3s ≠ kubeadm K8s (minor behavioural differences)
C

Namespaces + RBAC — no extra infrastructure

Use separate namespaces (prod, staging, test) with RBAC and NetworkPolicy for isolation within the same cluster.

Pros

  • Zero RAM cost
  • Simplest to manage

Cons

  • Not a real cluster boundary — CRDs, cluster-scoped resources are shared
  • Weak isolation for destructive integration tests
  • Less realistic for mock infra testing

My recommendation: A (vCluster) — you get a real K8s API surface for integration/e2e tests, costs almost nothing in RAM, and fits perfectly with the existing Helm-based setup. Keep prod at Option B.
